================================================================================
  СХЕМА РАБОТЫ АГЕНТОВ (UAE Chat / Streamlit + OpenRouter)
  Кратко и просто: кто что делает и в каком порядке
================================================================================

┌─────────────────────────────────────────────────────────────────────────────┐
│  ЧТО ЭТО ВООБЩЕ?                                                            │
│  • В чате: по желанию — блок «Welcome agents» в сайдбаре: имя, язык,         │
│    кнопка Start генерирует два коротких сообщения из сессионных JSON (см. ниже).│
│  • В чате: пользователь пишет сообщение. В сайдбаре выбран «режим» (интент).│
│  • При необходимости загружены два JSON-файла (session и steps) — они идут в │
│    нейросеть как данные для анализа. Все ответы идут через OpenRouter.       │
└─────────────────────────────────────────────────────────────────────────────┘


┌─────────────────────────────────────────────────────────────────────────────┐
│  ЧТО ТАКОЕ «АГЕНТ» ЗДЕСЬ?                                                   │
│  Это не отдельные программы. Это разные РОЛИ вызова модели через OpenRouter:│
│  • «Классификатор» (intent_classifier) — тип сообщения в чате.              │
│    Теперь различает 4 намерения: general_chat, session_question,             │
│    visualization_request, skill_creation.                                    │
│  • «Сессионный помощник» (session_question) — ответ по JSON в чате.          │
│  • «Визуализатор» (visualization_agent) — один вызов, две части:             │
│    текстовый ответ + chart_spec (JSON с осями и данными для графика).        │
│    Если данные не поддерживают график — chart_spec = null.                  │
│  • «Персонализация завершения» (completion_personalization) — в чат:         │
│    тёплое приветствие из контекста completion_rationale.                    │
│  • «Вовлечение в диалог» (session_engagement) — в чат: одна строка с          │
│    интересными фактами из session (+ steps), без приветствия и без имени;   │
│    чтобы склонить к дальнейшему разговору.                                    │
│  Разница — в тексте промпта (задание, правила) и в выбранной модели.         │
└─────────────────────────────────────────────────────────────────────────────┘


┌─────────────────────────────────────────────────────────────────────────────┐
│  ВХОДНЫЕ ДАННЫЕ (напоминание)                                                │
│  • API-ключ   — в Streamlit Secrets (OPENROUTER_API_KEY).                    │
│  • Session   — произвольный JSON из 1-го файла (идёт в контекст как «session»).│
│  • Steps     — произвольный JSON из 2-го файла (идёт как «steps»).         │
│  • Последние 6 сообщений диалога — для классификатора, сессионного агента    │
│    и визуализатора (все три получают recent_messages в промпте).            │
│  • Pending skill question — опционально: после ответа «поставить» ещё вопрос.│
│  • Блок Welcome: имя (только для агента персонализации completion), язык     │
│    вывода (english / german / ukrainian / russian).                          │
└─────────────────────────────────────────────────────────────────────────────┘


================================================================================
  ПОТОК «WELCOME» (КНОПКА START, ДО СООБЩЕНИЯ В ЧАТ)
================================================================================

  Условия:
  • Загружен валидный Session (JSON).
  • В JSON есть массив(ы) `goals`: берётся первая запись по порядку списка, где
    `goal_completion_status` равен «complete» (без учёта регистра),
    и из неё читается `completion_rationale` (непустая строка).
  • Если такой записи нет — Start показывает ошибку, вызовы LLM не выполняются.

  Порядок вызовов:

    ┌──────────────┐
    │  Пользователь  │
    │  нажимает    │

                   ▼
    ┌──────────────────────────────┐
    │  Агент 1: completion_        │  Модель по умолчанию: openai/gpt-4o-mini
    │  personalization             │  Выход JSON: {"personalized_message": "..."}
    │                              │  Вход: rationale, имя, код языка
    └──────────────┬───────────────┘
                   │  только если HTTP 200

================================================================================
  ЧЕТЫРЕ РЕЖИМА В САЙДБАРЕ (ПРОСТЫМИ СЛОВАМИ)
================================================================================

1)  session_question
    ────────────────
    «Я уже знаю, что вопрос относится к сессии / отчёту.»
    Нейросеть вызывается ОДИН раз — сразу «сессионный агент».
    Подходит, когда не хочется лишнего шага с классификатором.

2)  visualization_request
    ───────────────────────
    «Хочу увидеть график / сравнение / динамику.»
    Нейросеть вызывается ОДИН раз — «визуализатор» (build_visualization_messages).
    Ответ: текст + chart_spec JSON → Streamlit рисует bar или line chart
    прямо в сообщении. chart_spec хранится в message dict и
    перерисовывается при каждом rerun.

3)  general_chat
    ────────────
    «Определить тип вопроса автоматически.»
    Сначала — классификатор (intent_classifier). Возможные resolved-значения:
      • visualization_request → вызов визуализатора (_run_visualization)
      • session_question      → вызов сессионного агента
      • general_chat          → только метка в чате (без второго LLM)
      • skill_creation        → только метка (заглушка)

4)  skill_creation
    ───────────────
    «Режим для создания скилла» — в демо НЕ вызывается модель, только
    текст, что логику нужно настроить в бэкенде. Это заглушка.


================================================================================
  ГДЕ «ЖИВУТ» ПРОМПТЫ (В КОДЕ)
================================================================================

• Классификатор:              build_intent_classifier_messages
• Сессионный агент (чат):      build_session_messages
• Визуализатор:              build_visualization_messages
  – Один вызов; выход: {"assistant_answer": "...", "chart_spec": {...|null}}
  – chart_spec schema: {chart_type, title, x_label, y_label, x_values[], y_values[]}
  – Рендер: render_chart_spec → st.bar_chart или st.line_chart (pandas DataFrame)
• Персонализация completion: build_completion_personalization_messages
• Вовлечение (факты из сессии): build_session_engagement_messages
• Все HTTP-вызовы:          call_openrouter (совместимый с OpenAI /chat/completions
  через https://openrouter.ai/...; тело — JSON, токен в заголовке).

Модели в демо:
• Чат (выбор в сайдбаре «Model») — обычно из пресетов или custom.
• Welcome агент 1 — фиксировано openai/gpt-4o-mini (константа в коде).
• Welcome агент 2 — фиксировано google/gemini-2.5-pro (константа в коде).


================================================================================
  ДОПОЛНИТЕЛЬНО: Pending skill question
================================================================================

Если это поле в сайдбаре не пустое, в промпт сессионного агента добавляется
правило: после ответа на вопрос пользователя «мягко перейти» и задать
ещё один, указанный вами вопрос (сценарий вроде «продолжения skill»).

То же поле подхватывает и визуализатор (build_visualization_messages): после
текстового ответа и описания графика модель также должна встроить pending question.


================================================================================
  ФОРМАТ chart_spec И ОТОБРАЖЕНИЕ
================================================================================

• Поля: chart_type ("bar" | "line"), title, x_label, y_label, x_values[], y_values[].
• В коде parse_visualization_response принимает chart_spec только если x_values и y_values
  одинаковой длины и непустые; иначе график не показывается (chart_spec отбрасывается).
• В сообщении ассистента chart_spec хранится отдельным ключом в dict сообщения;
  текст assistant_answer остаётся в content для истории чата.
